# 聊天会话行为设置设计

## 背景

“聊天与通知”分组首版已经接入两个立即生效的发送侧设置：

- `Enter 发送消息`
- `广播前确认`

但消息接收后的主界面行为仍是固定的：

- 收到新的单聊消息时，只会把消息追加到状态中，不会自动切换到来源联系人
- 收到新的文件投递消息时，也不会自动切换到来源联系人

这意味着用户在当前正看其他会话时，虽然消息已经到达，但仍需要自己去联系人列表里定位发送者再切过去。

## 目标

- 让“聊天与通知”继续补充一档无需新系统依赖、可以立即生效的接收侧行为设置
- 允许用户决定收到单聊消息后，是否自动切到来源会话
- 允许用户决定收到文件投递消息后，是否自动切到来源会话
- 保持默认行为稳妥，不强行打断当前正在查看的会话

## 非目标

- 本次不引入系统桌面通知
- 本次不实现未读数、联系人红点或置顶排序
- 本次不改造广播消息的独立入口
- 本次不改造消息持久化结构

## 核心问题与设计修正

在方案收敛过程中，原本考虑过增加“单聊中显示广播消息”开关，但结合当前 UI 发现存在隐藏问题：

- 当前主界面总会自动选中一个在线联系人
- 广播消息没有独立会话或独立入口
- 如果关闭“在单聊中显示广播”，广播消息会变成“收到但无稳定查看入口”

因此本次不做该项开关，改为选择两个没有副作用、且能直接落地的会话自动切换设置。

## 方案

### 1. 收到单聊自动切换到来源会话

新增聊天偏好：

- `autoSwitchToIncomingDirect: boolean`

行为：

- 当主界面收到来自其他设备的 `direct` 消息时
- 如果该开关开启，并且来源设备仍在在线联系人列表中
- 则自动调用 `selectDevice(sourceDeviceId)`，把当前会话切到发送者

默认值：

- `false`

原因：

- 自动切会话会打断当前阅读，因此默认保持稳妥

### 2. 收到文件投递自动切换到来源会话

新增聊天偏好：

- `autoSwitchToIncomingDelivery: boolean`

行为：

- 当主界面收到来自其他设备的 `delivery` 消息时
- 如果该开关开启，并且来源设备仍在在线联系人列表中
- 则自动切换到发送者会话

默认值：

- `true`

原因：

- 文件投递请求往往需要立即接收/取消，自动切过去能更快看到操作卡片

### 3. 切换范围与约束

本次仅在 `view === "main"` 的消息事件监听中执行自动切换逻辑。

不会自动切换的场景：

- 当前是设置窗口
- 消息来自本机自己
- 来源设备已经不在联系人列表里
- 对应开关关闭

## 数据模型

### 前端

`src/desktop/types.ts`

- 在 `ChatPreferences` 中新增：
  - `autoSwitchToIncomingDirect`
  - `autoSwitchToIncomingDelivery`

`src/app/store.ts`

- 默认设置同步补齐这两个字段

`src/desktop/api.ts`

- `getRuntimeSettings()` 与 `syncRuntimeSettings()` 补充字段映射
- 对缺失字段继续使用默认值兜底

### Rust

`src-tauri/src/models.rs`

- 扩展 `ChatPreferences`：
  - `enter_to_send`
  - `confirm_before_broadcast`
  - `auto_switch_to_incoming_direct`
  - `auto_switch_to_incoming_delivery`

`src-tauri/src/settings_store.rs`

- 旧版配置迁移路径继续默认填充 `ChatPreferences::default()`

## 交互设计

设置页“聊天与通知”分组新增两个复选项：

- `收到单聊自动切换到来源会话`
- `收到文件投递自动切换到来源会话`

说明文案强调：

- 单聊自动切换适合把聊天窗口当作主监控入口的用户
- 文件投递自动切换默认开启，因为更贴近当前接收流程

## 测试策略

### 前端测试

在 `src/test/app-shell.test.tsx` 补充：

- 开启单聊自动切换后，收到来自 Bob 的实时单聊时，会话从 Alice 切到 Bob
- 关闭单聊自动切换时，收到来自 Bob 的实时单聊后，仍停留在 Alice 会话
- 开启文件投递自动切换后，收到来自 Bob 的投递请求时，会话切到 Bob 且能看到接收卡片
- 设置页能展示并更新这两个新开关

- `npm.cmd run test -- app-shell`
- `npm.cmd run test`
- `cargo check`

## 风险与权衡

- 自动切会话可能打断当前阅读，因此单聊自动切换默认关闭
- 文件投递自动切换默认开启，优先保证接收动作的可见性
- 由于本次不引入未读标记，关闭自动切换后仍然是“消息到达但用户需自行点击联系人查看”，这是下一阶段再解决的问题
