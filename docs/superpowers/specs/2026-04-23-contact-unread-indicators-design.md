# 联系人未读提示设计

## 背景

当前主界面已经支持：

- 联系人列表与当前会话联动
- 实时单聊消息接收
- 实时文件投递消息接收
- 聊天与通知里的会话自动切换设置

但当用户关闭“收到单聊自动切换到来源会话”后，新的单聊或文件投递虽然已经到达，联系人列表里仍没有任何显式提醒，用户必须自己逐个点击联系人才能确认来源。

## 目标

- 在不改造当前会话结构的前提下，为联系人列表补上一档最小可用的未读提示
- 让未当前选中会话的单聊消息和文件投递消息，能够在联系人列表上形成显式提醒
- 当用户点击对应联系人进入会话后，未读提示立即清除

## 非目标

- 本次不改造广播消息入口或广播会话形态
- 本次不做联系人置顶、排序重排或最近消息预览
- 本次不做系统通知、免打扰或提醒声音
- 本次不把传输任务面板并入会话区

## 方案对比

### 方案 1：联系人显示未读数字

做法：

- 联系人右侧显示一个未读数字徽标
- 收到来自该联系人的未读单聊或文件投递时累加
- 点击该联系人后清零

优点：

- 信息最直观，用户一眼能看到有没有积压
- 后续可平滑扩展到“待处理投递数”等更细粒度提示

缺点：

- 需要在前端状态层维护每个联系人的计数

### 方案 2：联系人只显示红点

做法：

- 联系人右侧只显示有无未读的圆点

优点：

- 实现更轻

缺点：

- 无法判断未读量
- 后续如果要补数量，大概率仍要改状态模型与 UI

### 方案 3：按最新消息自动置顶联系人

做法：

- 收到消息后，把对应联系人移到顶部

优点：

- 更接近传统 IM 列表

缺点：

- 会打乱当前稳定联系人顺序
- 会和现有“默认自动选中第一个联系人”的行为产生更大交互变化
- 超出本轮最小迭代范围

## 结论

本次采用方案 1：联系人显示未读数字。

原因：

- 当前主界面缺的不是更复杂的会话结构，而是一个明确、立即可见的提醒入口
- 数字提示在当前联系人数量不多的场景下信息密度更高，也更便于后续扩展

## 交互设计

### 1. 联系人列表中的显示方式

每个联系人卡片右上角新增一个未读数字徽标：

- 未读数为 `0` 时不显示
- 未读数大于 `0` 时显示对应数字

视觉上保持轻量，不改变当前联系人卡片的主信息层级。

### 2. 计数规则

仅统计以下消息：

- 来自其他设备的 `direct` 消息
- 来自其他设备的 `delivery` 消息

不统计：

- 本机自己发出的消息
- `broadcast` 消息
- 来自不在当前在线联系人列表中的设备消息

### 3. 清零规则

当用户显式点击某个联系人并切换到该会话时：

- 该联系人的未读计数立即清零

如果收到消息时当前已处于该联系人的会话：

- 不增加未读数

### 4. 与自动切换设置的关系

如果收到消息后因为设置开启而自动切到来源联系人：

- 该联系人不会保留未读提示

也就是说，未读提示主要补足“没有自动切过去”的场景，而不是和自动切换并行叠加。

## 状态设计

在前端 store 中新增联系人未读状态，按 `device_id` 维护：

- `Record<string, number>`

这样可以做到：

- 与消息持久化结构解耦
- 不影响 Rust 侧设置与消息模型
- 后续可以单独扩展清零、累加、批量同步逻辑

## 代码边界

### `src/app/store.ts`

- 扩展 store 状态，新增未读映射
- 新增递增未读与清除未读的方法
- 在选择联系人时清除该联系人的未读
- 在设备列表变化时清理已离线联系人的未读残留

### `src/App.tsx`

- 在 `chat-message-received` 事件里，根据当前选中联系人、消息类型和来源设备决定是否给联系人增加未读数
- 保持当前自动切换逻辑不变，仅在“未发生会话切换”时留下未读提示

### `src/desktop/modules/contacts/ContactsPanel.tsx`

- 接收未读映射
- 在联系人卡片中渲染未读数字徽标

### `src/styles/app.css`

- 为联系人卡片补充未读徽标样式

## 测试策略

在 `src/test/app-shell.test.tsx` 补充以下场景：

- 当当前停留在 Alice，会收到 Bob 的单聊时，Bob 显示未读数字
- 点击 Bob 联系人后，未读数字清除
- 当当前已在 Bob 会话里收到 Bob 的单聊时，不显示未读数字
- 收到广播消息时，不产生联系人未读提示
- 收到文件投递消息且未自动切换时，来源联系人显示未读数字

## 风险与权衡

- 未读状态本次只在前端内存中维护，应用重启后不会恢复；这符合当前“最小可用提醒”的定位
- 广播仍沿用当前“始终显示在聊天流里”的做法，不纳入联系人未读，避免所有联系人同时被污染
- 当前联系人列表没有分组与排序，本次只做加法式提示，不主动改变用户已熟悉的列表顺序
