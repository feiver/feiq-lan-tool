# 刷新历史详情回看设计

## 背景

当前“网络与发现”页已经具备最近刷新历史的摘要能力，包括：

- 成功 / 失败结果记录
- 本地持久化
- 一键清空
- 仅失败筛选
- 默认折叠与展开更多
- 失败项立即重试

但历史项目前仍然只保存摘要字段，用户只能看到：

- 新增发现数量
- 已在线数量
- 未命中网段数量
- 失败原因

真正用于判断问题的详情，例如：

- 那一次新增了哪些设备
- 哪些设备在刷新前已经在线
- 哪些手动网段仍未命中
- 每个网段是“本次新增命中 / 已有在线设备 / 当前未命中”

仍然只存在于“最近一次刷新结果”的临时状态里，无法随历史项一起回看，也无法在重启后恢复。

## 目标

- 让每一条刷新历史都能独立回看详情，而不是只保留摘要
- 让用户可以在历史列表中按需展开某一条记录，查看那次刷新的完整诊断快照
- 让刷新详情在应用重启后仍可恢复
- 保持最近刷新历史列表的紧凑度，不把所有详情默认铺开

## 非目标

- 本次不新增新的后端发现协议
- 本次不增加历史搜索、导出、分页或长期归档
- 本次不把历史上限从 5 条改大
- 本次不引入新的独立详情弹窗

## 核心决策

### 1. 历史项从“摘要记录”升级为“摘要 + 详情快照”

当前 `DiscoveryRefreshHistoryEntry` 只包含计数与失败文案，无法承载回看需求。

本次将历史项扩展为：

- 基础摘要
  - `id`
  - `timestamp`
  - `status`
  - `discoveredCount`
  - `existingCount`
  - `unmatchedSegmentCount`
  - `message`
- 详情快照
  - `newDeviceLabels`
  - `existingDeviceLabels`
  - `unmatchedSegments`
  - `segmentStatuses`

其中：

- 成功刷新时保存完整详情快照
- 失败刷新时详情数组为空，仅保留失败文案

这样历史列表中的任意一项，都能在脱离“最近一次刷新结果”状态的情况下独立展示。

### 2. 历史详情采用“列表内展开”而不是“默认铺开”或“独立弹窗”

采用列表内展开，原因是：

- 比默认铺开更紧凑
- 比弹窗更轻量
- 与当前设置页的诊断信息结构保持一致

交互规则：

- 每条历史默认折叠
- 点击 `查看详情` 展开
- 展开后按钮变成 `收起详情`
- 同一时间只展开 1 条历史项
- 过滤、清空历史或目标记录消失时，自动清除展开状态

### 3. 成功项和失败项展示不同层级的信息

成功项展开后展示：

- 新增发现设备
- 刷新前已在线设备
- 当前配置网段仍未命中
- 网段状态标签列表

失败项展开后展示：

- 失败原因
- 若详情快照为空，则不显示空列表，只显示说明

这可以让历史详情延续当前“最近一次刷新结果”的诊断心智，而不是重新创造一套展示模型。

### 4. 历史文件保持向后兼容

本地已经可能存在旧版历史 JSON。为了避免升级后旧数据无法读取：

- Rust 端历史模型新增字段必须带默认值
- 旧文件缺失详情字段时，自动读取为空数组
- 前端桥接层同样对缺失字段做兜底

这样旧历史仍可正常显示摘要，只是无法回看旧记录的详情。

## 数据模型设计

### 前端类型

`src/desktop/types.ts` 中的 `DiscoveryRefreshHistoryEntry` 扩展为：

- `newDeviceLabels: string[]`
- `existingDeviceLabels: string[]`
- `unmatchedSegments: string[]`
- `segmentStatuses: SegmentRefreshStatusSummary[]`

为避免循环依赖，`SegmentRefreshStatusSummary` 可继续保留在设置模块侧，或抽到共享类型中；实现时优先选改动面更小的方式。

### 前端运行时

`App.tsx` 中：

- 成功刷新后，创建历史项时把 `buildRefreshResultSummary()` 的结果一起写入历史
- 失败刷新后，创建历史项时写入空详情数组
- 读取历史后直接进入 `discoveryRefreshFeedback.history`

### Tauri 桥接

`src/desktop/api.ts` 中：

- `getDiscoveryRefreshHistory()` 需要映射新增详情字段
- `persistDiscoveryRefreshHistory()` 需要写出新增详情字段

### Rust 持久化

`src-tauri/src/models.rs` 中的历史结构体增加：

- `new_device_labels`
- `existing_device_labels`
- `unmatched_segments`
- `segment_statuses`

其中 `segment_statuses` 的元素需要包含：

- `segment`
- `status`
- `new_device_count`
- `existing_device_count`

所有新增字段都需要通过 `serde(default)` 保证旧文件可读。

## 界面与交互设计

### 历史列表

每条历史继续保留当前两层信息：

1. 时间 + 成功/失败状态 badge
2. 一行摘要文案

并新增第三层操作：

- 成功项：`查看详情`
- 失败项：`查看详情`
- 失败项继续保留现有 `立即重试`

### 展开内容

展开后在该历史项下方插入一个轻量诊断块，视觉风格延续当前设置页已有的 `settings-diagnostics-block`。

成功项内容：

- `新增发现设备`
- `刷新前已在线设备`
- `当前配置网段仍未命中`
- `网段状态`

各列表为空时展示明确文案，而不是空白区域。

失败项内容：

- `失败原因`

### 状态同步

以下场景需自动重置展开项：

- 切换“全部 / 仅失败”筛选
- 点击“清空历史”
- 历史被截断导致当前展开项不再存在

## 错误处理与兼容策略

- 旧版历史文件缺失详情字段：按空数组处理
- 单条历史的详情字段异常：该条退化为仅显示摘要，不影响整页
- 历史保存失败：沿用当前策略，不阻塞主流程

## 测试策略

### 前端测试

新增或调整 `src/test/app-shell.test.tsx`，覆盖：

- 成功历史项点击 `查看详情` 后能展开完整诊断内容
- 失败历史项点击 `查看详情` 后能看到失败原因
- 同一时间只展开 1 条历史
- 历史详情在重新加载设置页后仍能恢复

### Rust 测试

调整 `src-tauri/tests/settings_store_tests.rs`，覆盖：

- 带详情快照的历史能完整 roundtrip
- 旧格式历史 JSON 在缺失新增字段时仍能加载成功

## 验收标准

- 用户能在“最近刷新历史”中查看任意一条记录的详情
- 成功记录可回看新增设备、已有设备、未命中网段和网段状态
- 失败记录可回看失败原因
- 历史详情重启后仍存在
- 旧版历史文件不会因字段升级而读取失败
