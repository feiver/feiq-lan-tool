# 原生文件选择器设计

## 背景

当前桌面端“发送文件”流程依赖用户手动输入本地文件路径。这个交互虽然能跑通链路，但不符合 FeiQ 类局域网工具的常见使用习惯，也明显弱于桌面应用应有的可用性。

本次改动只解决“如何选择待发送文件”这一件事，不改文件发送协议、不改传输状态模型、不改接收链路。

## 目标

- 将“手动输入文件路径”升级为“点击按钮调用系统文件选择器”
- 保留现有发送逻辑，选中文件后仍复用 `sendFileToDevice(...)`
- 保持跨平台能力，基于 Tauri 2 官方 dialog 插件实现
- 为后续“下载目录原生选择器”预留一致的技术路径

## 非目标

- 不支持多文件选择
- 不支持目录选择
- 不新增最近文件、历史记录或拖拽上传
- 不调整 Rust 文件传输协议和发送实现
- 不在本次改动中处理发送失败提示弹窗

## 方案选择

### 方案 A：Tauri 原生 dialog 插件

前端使用 `@tauri-apps/plugin-dialog` 的 `open(...)` 打开系统文件选择器；Rust 侧注册 `tauri_plugin_dialog::init()`；capability 增加 `dialog:default`。

这是推荐方案，因为它是 Tauri 2 官方路径，跨平台一致、代码量最小，也最容易复用到后续目录选择场景。

### 方案 B：自定义 Tauri command 包装原生选择器

不采用。它会重复封装官方已提供的能力，增加维护面，但不会带来实际收益。

### 方案 C：继续保留文本输入，只做辅助校验

不采用。它不能真正解决桌面体验问题，只是改善现有低效交互。

## 交互设计

聊天区文件发送入口调整为：

- 一个只读文本框，用于展示当前已选择文件路径
- 一个“选择文件”按钮，用于打开系统文件选择器
- 一个“发送文件”按钮，沿用当前发送动作

行为规则：

1. 未选择联系人时，“选择文件”与“发送文件”都禁用
2. 点击“选择文件”后：
   - 选中文件：更新展示路径
   - 取消选择：保持当前路径不变，不报错
3. 点击“发送文件”后：
   - 仍调用现有 `sendFileToDevice(addr, filePath, toDeviceId)`
   - 成功后清空当前已选路径
4. 不再允许手工编辑路径，避免出现无效路径或误输入

## 技术设计

### 前端

- 在 `src/desktop/api.ts` 新增文件选择方法，例如 `pickFilePath()`
- 在 `src/App.tsx` 中新增选择文件处理函数，负责调用 `pickFilePath()` 并更新 `draftFilePath`
- 在 `src/desktop/modules/chat/ChatPanel.tsx` 中调整文件发送 UI：
  - 输入框改为只读
  - 新增“选择文件”按钮
  - 保留“发送文件”按钮
- 在 `src/styles/app.css` 中补充文件选择区样式，使两个按钮与只读路径展示保持一致视觉层级

### Tauri / Rust

- `package.json` 增加 `@tauri-apps/plugin-dialog`
- `src-tauri/Cargo.toml` 增加 `tauri-plugin-dialog`
- `src-tauri/src/main.rs` 注册 `tauri_plugin_dialog::init()`
- `src-tauri/capabilities/default.json` 增加 `dialog:default`

本次不新增 Rust command，因为文件选择器完全可以由插件直接提供。

## 错误处理

- 用户取消选择文件时视为正常交互，不提示错误
- 若 dialog API 调用失败，本次先保持静默失败，不清空已有路径
- 若未选择联系人，则不能打开文件选择器，从交互层直接禁用按钮

## 测试策略

前端以 TDD 方式补两类测试：

1. 点击“选择文件”后，mock 的 dialog 返回路径，界面应展示该路径
2. dialog 返回 `null` 表示取消选择时，界面保持原状态，不触发发送

回归验证：

- 现有 `app-shell` 测试全部通过
- `npm.cmd run build` 通过

本次不新增 Rust 单元测试，因为改动只涉及插件注册和 capability 配置，不包含新的 Rust 业务逻辑。

## 验收标准

- 用户无需手输路径即可选择单个本地文件
- 取消选择不会污染当前状态
- 已选文件仍能通过现有发送链路发送给当前联系人
- 现有消息、传输、设置相关测试不回归
