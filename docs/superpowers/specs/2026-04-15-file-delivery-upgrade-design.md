# 文件投递升级设计

## 背景

当前桌面端文件发送已经具备基础单文件传输能力，但整体交互仍停留在 MVP 早期形态：

- 发送方需要手工输入文件路径
- 接收方没有“接收 / 取消”的明确决策入口
- 目录、多文件、拖拽上传都还不支持
- 文件消息与真实传输任务之间缺少更清晰的两阶段关系

本次设计目标不是只补一个文件选择器，而是把“文件发送”升级为更接近桌面局域网工具直觉的“文件投递”流程。

## 目标

- 支持发送单文件、多文件、目录、拖拽上传
- 发送方先发出投递请求，接收方明确 `接收 / 取消`
- 接收方点击 `接收` 后，再选择保存目录
- 接收完成后，在文件消息下提供 `打开目录`
- 目录发送时保留原始相对目录结构
- 为后续“群组发送”保留可复用的投递模型

## 非目标

- 本次不实现群组发送本身
- 本次不实现离线补投
- 本次不实现断点续传
- 本次不实现目录打包压缩后发送
- 本次不实现上传 / 接收速度限制与高级设置

## 核心决策

### 1. 采用两阶段投递

文件投递拆为两个阶段：

1. 发送方发出投递请求，只携带投递元数据，不立刻传文件字节
2. 接收方确认接收并选定保存目录后，再开始真实传输

这样可以让接收方拥有更强控制权，也能自然承载“取消”“另存位置”“完成后打开目录”等桌面交互。

### 2. 混合内容采用混合展示

当发送方选择“文件夹 + 若干散文件”时，界面不合并成一个总批次，而采用混合展示：

- 目录以目录组展示
- 散文件逐个单独展示

例如：

```text
投递内容
- 项目资料/   （目录组）
- 报价单.xlsx （单文件）
- 封面.png    （单文件）
```

目录组内部在发送元数据层面仍包含所有子文件及其相对路径。

### 3. 接收端以用户选择目录作为接收根目录

接收方点击 `接收` 后弹出“选择保存目录”。用户选定的目录作为接收根目录，目录内部结构在此根目录下重建。

例如发送方投递：

```text
项目资料/
  图片/logo.png
  文档/说明.txt
```

接收方若选择：

```text
D:\接收区
```

则落盘结果为：

```text
D:\接收区\项目资料\图片\logo.png
D:\接收区\项目资料\文档\说明.txt
```

### 4. 请求默认一直等待

发送方发出投递请求后，若接收方暂不操作，请求默认一直等待，不自动超时取消。

这更符合即时通信工具中“等待对方处理”的心智模型。

## 用户交互

### 发送方

#### 发起投递

发送方可通过三种方式选择内容：

- 点击按钮选择单个或多个文件
- 点击按钮选择目录
- 将文件、目录或混合内容拖拽到聊天区域

选择完成后，聊天区展示待发送内容摘要：

- 散文件逐个显示
- 目录以目录组显示

发送方点击“发送”后，生成一条文件消息卡片并进入“等待对方处理”状态。

#### 等待阶段

在等待阶段，发送方消息卡片显示：

- 投递内容摘要
- 当前状态：等待接收
- 若对方取消：显示“对方已取消”
- 若对方接收后开始传输：显示“对方已接收，开始传输”

### 接收方

#### 收到投递请求

接收方在消息区看到一条文件消息卡片。卡片内容包括：

- 发送方
- 投递摘要
- 若包含目录，则显示目录组
- 操作按钮：`接收`、`取消`

#### 点击取消

接收方点击 `取消` 后：

- 本地消息状态变为“已取消”
- 回馈发送方“对方已取消”
- 本次请求结束，不再进入传输阶段

#### 点击接收

接收方点击 `接收` 后：

1. 弹出系统目录选择器
2. 用户选择一个保存目录作为接收根目录
3. 选择完成后正式进入数据传输阶段

若用户在目录选择器中取消：

- 保持该请求仍处于“待处理”状态
- 不自动视为“取消”

#### 传输完成

接收完成后，消息卡片显示：

- 完成状态
- `打开目录` 按钮

点击 `打开目录` 后，使用桌面能力打开接收根目录。

## 数据模型设计

### 发送内容模型

投递请求不再只是一个文件路径，而是一个内容集合：

- 单文件项
- 多文件项
- 目录组

目录组包含：

- 目录显示名
- 目录下子文件清单
- 每个子文件相对该目录根的相对路径

### 消息模型

聊天消息需要扩展新的“文件投递消息”语义，至少覆盖以下状态：

- `PendingDecision`：等待接收方处理
- `Accepted`：接收方已接收，等待或进入传输
- `Rejected`：接收方取消
- `InProgress`：传输中
- `Completed`：传输完成
- `Failed`：传输失败

该状态用于消息卡片展示；传输面板仍保留文件级任务进度。

### 传输模型

真实字节传输仍按文件级执行，不按目录整体打包。

因此：

- 一个散文件对应一个传输任务
- 一个目录组会展开成多个文件传输任务

目录组只是消息视图和发送内容视图上的逻辑分组，不改变底层单文件传输事实。

## 协议设计

现有协议需要从“立即发送文件”升级成“请求 - 决策 - 传输”三段式。

建议新增三类事件：

1. 投递请求事件
内容包括：
- 请求 ID
- 发送方
- 投递内容清单
- 目录组及子文件相对路径

2. 投递响应事件
内容包括：
- 请求 ID
- 响应结果：接收 / 取消
- 若接收，可附带进入传输阶段的确认信息

3. 文件传输事件
沿用现有文件传输能力，但与请求 ID / 内容项建立映射关系

这样后续扩展到群组时，只需要把“目标”从单个联系人换成多个在线成员，而不需要重写整套文件投递语义。

## 前端设计

### 聊天区

新增统一投递入口：

- `选择文件`
- `选择文件夹`
- 拖拽投递区域

发送前展示待投递内容摘要。

### 消息卡片

新增文件消息卡片组件，支持：

- 展示散文件
- 展示目录组
- 在接收方侧展示 `接收 / 取消`
- 在完成后展示 `打开目录`

### 传输面板

传输面板继续显示文件级进度，但应能和消息卡片建立对应关系。目录组中的子文件任务可以按现有任务列表先逐条显示，后续再考虑折叠分组。

## 桌面能力设计

需要接入三类 Tauri 原生能力：

- 文件选择器：选择单个 / 多个文件
- 目录选择器：发送方选择目录、接收方选择保存目录
- 打开目录：接收完成后打开保存目录

这些能力应尽量通过官方插件实现，不额外发明复杂桥接。

## 错误处理

- 接收方取消投递：发送方明确看到“对方已取消”
- 接收方在选择保存目录时关闭目录选择器：请求仍保留为待处理
- 目录遍历失败：发送方在发起前就应阻止发送并提示失败
- 某个子文件传输失败：目录组整体状态显示“部分失败”，同时文件级任务保留失败明细
- 打开目录失败：只影响完成后的附加动作，不影响已完成状态

## 验收标准

- 发送方可发送单文件、多文件、目录和拖拽内容
- 接收方收到明确的文件消息卡片，并可 `接收 / 取消`
- 接收时必须先选择保存目录
- 目录内容在接收根目录下保留原始相对结构
- 接收完成后可从消息卡片直接 `打开目录`
- 发送方可以收到对方取消反馈
- 现有联系人、聊天、传输、设置功能不回归

## 后续扩展点

本设计明确为群组发送预留扩展口：

- 当前投递目标先是单个联系人
- 后续群组发送时，只需要把同一份投递请求分发给群组内在线成员
- 群消息层再额外叠加“已发给 X 人，Y 人离线”的结果汇总即可
