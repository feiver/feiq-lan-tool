# 联系人待接收投递提示设计

## 背景

当前主界面已经具备两层与消息相关的联系人提示能力：

- 联系人未读数字提示
- 会话区中的文件投递卡片，支持“接收 / 取消”

但当用户没有停留在对应联系人会话时，虽然新的文件投递请求已经到达，联系人列表里仍然无法明确区分：

- 只是有未读消息
- 还是有需要用户立刻决策的文件投递请求

这会导致用户需要先点进会话，才能知道是否存在待处理的投递动作。

## 目标

- 在联系人列表中单独标记“当前有待接收文件请求”的联系人
- 让“未读数字”和“待接收提示”并存，而不是互相覆盖
- 保持接收 / 取消动作仍在会话中的文件投递卡片里完成

## 非目标

- 本次不把“接收 / 取消”按钮移到联系人列表
- 本次不新增联系人排序、置顶或分组
- 本次不把已完成、已取消、已失败的投递长期保留在联系人提示中
- 本次不引入新的 Rust 状态模型或持久化结构

## 方案对比

### 方案 1：单独显示“待接收”标记

做法：

- 联系人卡片保留未读数字
- 如果该联系人存在入站且仍为 `PendingDecision` 的文件投递，再额外显示一个固定文案标记，如“待接收”

优点：

- 用户可以明确区分“有新内容”和“有待决策文件请求”
- 与当前会话卡片里的“接收 / 取消”职责分层清晰

缺点：

- 联系人卡片需要同时容纳两类提示，样式上要控制轻重

### 方案 2：把待处理投递也算入未读数字

做法：

- 不额外显示投递状态
- 只让未读数字继续累加

优点：

- 实现最轻

缺点：

- 无法体现“这是一条需要处理的投递请求”
- 用户仍需要点进会话确认是否需要接收 / 取消

### 方案 3：只显示待接收，不显示未读数字

做法：

- 用“待接收”覆盖联系人现有的未读感知

优点：

- 更强调文件请求的优先级

缺点：

- 会损失普通消息的提醒能力
- 与已落地的未读数字形成行为回退

## 结论

本次采用方案 1：未读数字与“待接收”标记并存。

原因：

- 文件投递请求的优先级通常高于普通消息，但它不应该吞掉普通消息提醒
- 分开显示可以让联系人列表成为“消息提醒 + 待处理动作提醒”的双入口

## 交互设计

### 1. 联系人卡片中的显示方式

联系人卡片右上区域继续显示未读数字，同时在卡片内容区增加一个轻量状态标记：

- 未读数字：沿用现有样式
- 待接收标记：文案固定为 `待接收`

如果某联系人同时存在这两种状态，则允许并存显示。

### 2. 判定规则

仅对以下场景显示“待接收”：

- 消息类型为 `delivery`
- 消息来自其他设备
- `delivery.status === "PendingDecision"`

不显示“待接收”的场景：

- 本机自己发出的投递
- 已接收、已取消、传输中、已完成、失败、部分失败的投递

### 3. 清除规则

“待接收”提示的生命周期与投递状态绑定，而不是与联系人点击行为绑定：

- 点击联系人进入会话：不移除“待接收”
- 在会话中点“接收”或“取消”后，如果该请求状态不再是 `PendingDecision`，则提示消失
- 如果同一联系人仍有其他待处理投递请求，则继续保留“待接收”

## 状态设计

本次不新增新的持久化模型。

由前端在主界面内根据现有 `messages` 推导联系人待接收状态：

- 读取所有 `delivery` 消息
- 只保留“来自其他设备”的入站请求
- 以 `request_id` 为粒度判断当前最新状态
- 汇总出仍然处于 `PendingDecision` 的来源设备集合

输出形态：

- `Record<string, boolean>` 或 `Set<string>` 均可

推荐直接在 `App.tsx` 中派生后传给 `ContactsPanel`，避免重复写入 store。

## 代码边界

### `src/App.tsx`

- 基于 `messages` 计算每个联系人是否存在待接收投递
- 将结果作为 props 传给 `ContactsPanel`

### `src/desktop/modules/contacts/ContactsPanel.tsx`

- 接收联系人待接收状态
- 在联系人卡片中渲染“待接收”标记

### `src/styles/app.css`

- 为联系人状态标记补充样式
- 保证未读数字和“待接收”并存时不会挤压主信息

## 测试策略

在 `src/test/app-shell.test.tsx` 补充以下场景：

- 收到来自 Bob 的入站 `PendingDecision` 投递时，Bob 联系人显示“待接收”
- 点击 Bob 联系人进入会话后，“待接收”仍然保留
- 对该投递点“接收”后，“待接收”消失
- 对该投递点“取消”后，“待接收”消失
- 当联系人同时有未读数字和待接收投递时，两者都能显示

## 风险与权衡

- 如果同一 `request_id` 在消息列表里存在多条历史记录，本次需要按“最新状态”聚合，避免旧的 `PendingDecision` 把已处理请求误判为待接收
- 本次只做联系人级布尔提示，不显示待处理投递数量；这是为了保持联系人卡片足够轻量
- 由于提示来自前端运行时推导，应用重启后会依赖现有消息列表重新计算，而不是独立持久化
